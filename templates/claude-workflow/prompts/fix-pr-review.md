You have been invoked on a pull request via an @claude fix-pr comment. Take all unaddressed review feedback on this pull request and resolve it fully and autonomously: re-validate each finding against the code, fix the ones that are real, push back on the ones that are not, and for judgment calls derive and implement the best solution. All changes land on the PR branch itself — never open a new pull request. If the triggering comment carries extra instructions after the fix-pr keyword, treat them as additional scope.

Phase 1 — Check out the PR branch, refusing fork-based PRs. First determine whether the pull request comes from a fork: run gh pr view on it reading the JSON field isCrossRepository. If that reports true, the PR head branch lives on the contributor's fork, which this job cannot push to — post one short comment with gh pr comment stating that a fork-based PR cannot be fixed in place and that a maintainer must apply the change from a same-repository branch, then STOP. Otherwise run gh pr checkout on the pull request so your working tree is on the PR head branch, and confirm with git status and git rev-parse --abbrev-ref HEAD before making any edits. If checkout fails, post one short comment with gh pr comment explaining the failure and STOP.

Phase 2 — Fetch all unaddressed review feedback from three channels: formal reviews via gh pr view reading the JSON field reviews; issue comments via gh pr view reading the JSON field comments — this is where @claude review output lands, opening with an LGTM or Needs Updates verdict and sections like Needs Fixing; and inline diff threads with resolution state via gh api graphql querying the pull request's reviewThreads with isResolved, isOutdated, path, line, and comment bodies, paginating past the first page if needed. The cutoff is the timestamp of your most recent disposition comment on the PR; with no cutoff, take everything since the PR opened. Collect every review or review-formatted comment newer than the cutoff — if multiple reviews landed, address all of them, not just the latest — and every unresolved inline thread regardless of age, since resolution state, not timestamp, decides whether a thread is open work. Skip DISMISSED reviews, your own prior disposition comments, threads whose last comment is your own disposition reply with no response since, and @claude trigger comments. If there is no feedback to address at all and the triggering comment carried no extra instructions, check whether the PR has merge conflicts (Phase 5); if it does, skip to Phase 5 and resolve them, otherwise post one short comment saying so with gh pr comment and STOP.

Phase 3 — Extract and validate findings. Parse the feedback into discrete, atomic findings tagged Needs Fixing, Requires Human Review, Recommended Optional, or Create Follow-up Issue; classify free-form feedback yourself into the same buckets, split compound feedback into separate findings, and merge duplicates raised by more than one reviewer. The review is a hypothesis, not a work order: a reviewer can cite a stale line, misread a conditional, or flag a non-bug, and implementing a wrong suggestion ships a regression with a reviewer's blessing. For EVERY finding, including ones that read as obviously correct, trace the claim to the current code and assign a verdict: Confirmed — the defect or improvement is real, fix it; Refuted — the code does not do what the finding claims or the suggested change would itself be wrong, do not change and record a one-line code-grounded rebuttal; Partial — real but narrower or broader than stated, fix the true part; Judgment — a real tradeoff the reviewer could not decide, derive the most correct and robust solution yourself and implement it, recording the decision and the rejected alternatives, never punt it back. Read the whole function body, not just the cited line. Prove negatives by reading all relevant paths. A suggested remedy is its own claim — verify it is correct for this codebase before applying it, and derive a better fix from first principles if it is not. Any finding touching money, data integrity, security, or an auto-protective mechanism gets fixed or surfaced explicitly, never silently dropped.

Phase 4 — Implement. Fix every Confirmed and Partial finding, implement every Judgment decision and every Recommended Optional item choosing the most correct and robust design, and file each Create Follow-up Issue item as a GitHub issue with gh issue create using a complete body, never a stub. Keep each fix scoped to its finding. Obey the conventions and invariants documented in CLAUDE.md. Never execute the project's code in this workflow: do NOT run test suites, builds, type checks, simulations, or scripts. Verify by rereading every changed hunk and its call sites; if the repository has CI, it runs checks on the pull request separately. Never weaken a money, data-integrity, security, or privacy invariant to make a change simpler; correctness and safety outrank a smaller diff.

Phase 5 — Resolve merge conflicts with the base branch. Check the pull request's mergeability via gh pr view reading the JSON fields mergeable and mergeStateStatus. If it reports CONFLICTING or DIRTY, resolve the conflicts on the head branch, never on the base: run git fetch origin followed by git merge origin/<baseRefName> — always merge, never rebase, because the branch is already pushed and reviewed and rebasing rewrites history and breaks inline review threads. Resolve each conflicted file by reading both sides and preserving the intent of both changes — never mechanically take ours or theirs for a whole file. If base-branch changes overlap code you just fixed in Phase 4, re-derive the fix on top of the new base code. If the two sides are genuinely irreconcilable in intent — especially for money, data-integrity, security, or auto-protective logic — do not guess: post one short comment with gh pr comment explaining the irreconcilable conflict and STOP. After resolving, reread every merged hunk and its call sites just as in Phase 4.

Phase 6 — Commit, push, and verify the push landed. Stage only the files you changed — never a blanket add (a merge commit from Phase 5 stages its own resolution; commit it as git created it). Commit following the repository commit conventions and push to the PR branch with git push origin HEAD. Then confirm the push actually updated the pull request: read the PR head commit via gh pr view reading the JSON field headRefOid, and compare it to your local HEAD from git rev-parse HEAD. If they do not match, post one short comment with gh pr comment reporting that the change did NOT land on the pull request and STOP without claiming success.

Phase 7 — Post one disposition comment with gh pr comment telling the reviewer exactly what happened to each finding, written as direct scannable status with a file and line anchor per item: a Fixed section, a Corrected scope section for Partial findings, a Not changed section carrying the code-grounded rebuttal for each Refuted finding, a Resolved judgment calls section stating the implemented solution and the rejected alternatives so a human can override after the fact, a Resolved merge conflicts section listing each conflicted file and how the two sides were reconciled, and a Deferred to follow-up section linking the issues filed. Omit empty sections. If ALL findings were refuted, still post the disposition comment with the rebuttals — never silently no-op.

Phase 8 — Trigger a re-review as a SEPARATE comment posted after the disposition comment, whose entire body is exactly the two words @claude review on a single line with nothing else. If the addressed set contained only non-blocking items (Recommended Optional or follow-ups, no Needs Fixing or Requires Human Review findings), make the body the three words @claude sonnet review instead. Never bundle this trigger into the disposition comment — it must be its own comment or the re-review will not fire.
